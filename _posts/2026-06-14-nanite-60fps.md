## 문제

Nanite는 폴리곤 수를 줄이지 않아도 된다는 인상을 주지만, 실제 프레임 시간은 더 복잡한 이유로 흔들린다. 메시가 Nanite로 들어갔는지보다 중요한 것은 화면에서 얼마나 많은 클러스터가 살아남고, 머티리얼과 오버드로우가 어느 지점에서 비용을 쓰는지였다.

> 폴리곤 수만 보고 판단하면 병목을 잘못 짚기 쉽다.

## 확인한 것

RenderDoc 캡처를 열어보면 Base Pass 이전 단계에서 이미 비용이 갈린다. 특히 큰 메시 하나보다 화면 곳곳에 흩어진 작은 Nanite 메시가 많을 때, 컬링과 가시성 판단 비용이 기대보다 커질 수 있다.

```cpp
// 예시: 실제 병목을 보기 전까지 LOD 숫자만 줄이지 않는다.
if (Mesh.IsNaniteEnabled())
{
    ProfileClusterVisibility(Mesh);
    ProfileMaterialCost(Mesh);
}
```

## 정리

우선순위는 단순했다.

- 먼저 GPU 캡처에서 비용이 생기는 패스를 확인한다.
- 폴리곤 수보다 머티리얼, 오버드로우, 인스턴스 분포를 같이 본다.
- Nanite를 끄는 판단은 마지막에 한다.

60fps를 지키는 일은 기능을 켜고 끄는 문제가 아니라, 비용을 정확히 보고 줄이는 순서의 문제에 가깝다.
